面试总结

找完工作也有一段时间了，感觉自己在前一段时间的面试表现，实在是有点大问题。最近也是希望能够做一些反思，对于后续的工作、找工作总结一些经验。

1. 自我介绍

  1. 基本信息：自我介绍时提供一些自己的基本信息，包括姓名、学历、工作经验、任职公司等。这些信息能够给面试官一个初步的了解。

  2. 技术经验：介绍一些技术经验，特别是与 Java 相关的经验。包括开发的项目、使用的技术栈以及在这些项目中承担的角色和责任。我们应该准备好对过去的工作经验中所做的项目、技术方案等进行详细说明。

  3. 发展路径：介绍一些学过的课程、证书，介绍自己的项目实战经验，包括工作内和工作外的。以及如何持续学习。

  4. 沟通能力：包括表达和表达意图的能力。应该能够清晰地陈述自己的观点和思路，能够与面试官进行良好的交流和沟通。

  5. 介绍项目：

2. 介绍项目

在自我介绍完后，面试官一般会让你去选择一个项目进行介绍。我自己是提供了两个项目，一个是偏技术的，一个是偏业务的。这样子面试官也会根据他的侧重点，去挑选一个项目去了解自己。

  1. 这个时候，一定要把握住节奏。最重要的是，要能够讲清楚简历上的项目。

    1. 项目概述：开门见山地介绍该项目的背景和目标。重点突出项目的重要性、目标以及你在项目中担任的角色。
    2. 技术栈和工具：说明你在项目中所使用的技术栈和工具，具体提及了哪些编程语言、框架、库以及版本控制系统等。这可以展示你对技术的熟悉程度和适应能力。
    3. 贡献和责任：说明你在项目中所承担的角色和职责，以及你为项目做出的贡献。可以提及你在设计、开发、测试、部署等方面的具体工作，以及你在解决技术问题和优化代码方面的能力。
    4. 成果和效益：强调你在该项目中所取得的成果和效益，例如项目的成功交付、用户反馈的积极响应、性能优化带来的效率提升等。
    5. 挑战和解决方案：提及在项目中遇到的挑战和困难，并说明你是如何解决这些问题的。这可以展示你的解决问题的能力和应变能力。（提前准备好一个解决过的问题，最好是 系统调优、数据库死锁 、 线上问题排查等等一些能够体现 技术力和思考能力 的问题。 ）
    6. 学习和发展：总结在该项目中所学到的经验和技能，并强调你持续学习和追求进步的态度。
    7. 涉及性能相关的数字： QPS 、TPS 、Redis 大小、机器数量、分库分表数量和原因、熔断/限流策略和阈值。
      1. 如何回复这些数字相关的问题呢：
        1. 如果有实际的线上数值，提供实际的数值。如果没有的话，以4C8G为例，提供一个压测环境的数字大小
        2. 描述时，一定要以一个维度去讲。比如说，我单台机器QPS 多少， 集群的QPS 多少。 限流/熔断 单台机器的限制阈值是多少。 峰值的QPS 多少，限流多少。
        3. 考虑28原则。 在20%的时间内，有80%的流量会打入。QPS、限流策略都是以80%去进行设计。
    8. 如果是偏业务的项目，讲清楚业务架构和业务流程。比如说用到DDD，要讲清楚DDD如何划域以及各个域的交互逻辑，还有每个域的聚合根。（体现出你对系统的了解）
    9. 如果是偏技术的项目，讲清楚系统架构和系统的目的。使用到的技术点。一定要想清楚性能数字。

  2. 重点强调与面试职位相关的技术和经验，把握好面试节奏，能够将面试官带入到你的节奏中，要不然就会很被动！！ （这次面试是我最为被动的面试，自己做的也不是很好，只能不断去回复面试官的问题。）

  3. 多用体现专业性的词语。

3. 遇到的问题

主要讲一些能够体现自己技术能力，或者思考能力的案例。比如说一些

  1. 内存泄漏的案例
  2. MQ消费延迟案例
  3. 机器重启丢失消息

使用好star方法分析问题

  1. 情境（Situation）：描述你所面临的具体情境和问题。解释清楚所面对的挑战、目标或需要解决的问题。

  2. 任务（Task）：明确你在该情境中承担的任务和责任。强调你所面对的具体目标和期望。

  3. 行动（Action）：说明你采取了什么具体的行动来应对情境和完成任务。描述你所采取的策略、方法、技术或行动计划。

  4. 结果（Result）：叙述你的行动带来的具体结果和成果。强调你在完成任务和解决问题方面取得的成就、效益或教训。如果可能的话，使用具体的数据、指标或评价来支持结果。

1. 在XX发现了什么问题，如何排查这个问题的，这个问题的根本原因是什么
2. 如何解决这个问题，单机验证、批量上线观察
3. 收获了什么成果或者教训

4. 场景设计题

如何应对场景设计题。（逐步更新）

1 以三方请求为例。

  1. 沟通好需求，题目能够提供啥，有没有什么限制。
  2. 同步的是啥类型的数据？就提供个接口给我们调？
  3. 存储设计和中间件选型，如果能丢mq可以异步解耦，可靠性会好点。如果不能，那就加重试去掉他们接口。
  4. 重试几次后触发告警，手工干预
  5. 设计个数据对账，让对方提供查询接口，定时核对数据。
